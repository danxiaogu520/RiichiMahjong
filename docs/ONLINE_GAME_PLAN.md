# 联网日麻游戏工程计划

## 1. 项目目标

将当前以 Rust 本地进程和 channel 驱动的日麻项目，逐步发展为一个可以和朋友稳定游玩的联网半庄游戏：

- 玩家通过房间码加入房间。
- 四名玩家可以准备并开始一局半庄。
- 独立服务器作为唯一权威，负责牌山、状态、合法性和结算。
- 客户端通过 WebSocket 接收状态和事件、提交玩家行动。
- 首先支持 Tauri 桌面客户端，同时保留浏览器开发预览能力。
- 暂不实现复杂账户、好友、匹配、排位和持久化战绩系统。

## 2. 当前基础与边界

当前项目已经具备以下核心能力：

- `riichi-core`：牌、手牌、牌山和玩家等基础领域类型。
- `riichi-logic`：向听、和牌、役种、符数、点数及牌效分析。
- `riichi-engine`：`GameState`、行动合法性、回合推进、响应窗口和结算。
- `riichi-proto`：客户端/服务端消息、玩家视角状态和隐藏信息过滤。
- `riichi-server`：网络、房间和连接边界，以及 session 到线协议的转换。
- `riichi-debug`：终端调试客户端，提供本地 AI 填位模式。
- `riichi-session`：传输无关的单房间游戏会话，承载 GameSession、内部命令和事件。

当前主要缺口是：协议兼容性收尾、完整断线恢复、Tauri 牌桌交互完善和部署流程。

当前进度（2026-07-14）：

- 阶段 4 已完成第一版 HTTP/WebSocket 实时传输闭环。
- 阶段 5 已创建 `tauri-app/`，完成 Tauri v2 工程骨架、前端依赖锁定、HTTP/WebSocket 传输层、房间大厅和牌桌占位界面。
- 浏览器开发预览已支持跨域联调；Tauri 原生窗口仍需在具备 GTK/WebKit 系统依赖的环境中完成编译验证。
- 阶段 5 第一轮牌桌 UI 已完成：可渲染玩家视角手牌、四家牌河、宝牌、点数和当前行动玩家；已接入出牌、自摸、立直、九种九牌以及鸣牌/荣和操作按钮和倒计时。
- `tauri-app` 的生产构建和 Rust workspace 全量测试均已通过。
- 阶段 6 第一轮已完成：WebSocket 断线采用指数退避自动重连；新连接从序号 0 和玩家视角快照恢复；鸣牌按钮会提交服务端提供的具体手牌组合，非法手牌不再显示为可操作状态。
- `RequestSnapshot` 已接入服务端事件队列，会返回请求玩家视角的最新可用状态更新，不会绕过隐藏信息投影。
- 客户端已支持从 `ActionRequired` 的候选列表选择立直弃牌、暗杠和加杠参数，不再默认提交第一个候选项。
- `riichi-session` 已增加重连回归测试，确认替换连接后会重新发送玩家视角状态和当前行动提示。

## 3. 目标架构

```text
tauri-app / web-client
        │ WebSocket + HTTP
        ▼
riichi-server
  ├── transport       HTTP、WebSocket、心跳
  ├── session         连接、座位、重连
  ├── room            房间码、入座、准备
  ├── application     房间和游戏服务编排
  └── game            GameSession、事件分发、超时
        │
        ▼
riichi-engine
        │
        ├── riichi-core
        └── riichi-logic
```

依赖方向必须保持单向：

```text
tauri-app  ───────► riichi-proto
web-client ───────► 协议定义
riichi-server ────► riichi-engine + riichi-proto
riichi-engine ────► riichi-core + riichi-logic
```

客户端不应依赖 `riichi-server` 或 `riichi-engine`。`riichi-debug` 通过 `riichi-session` 运行本地模式，未来网络模式只依赖协议和传输适配器。

## 4. MVP 范围

### 必须完成

- 创建房间并生成短房间码。
- 玩家以昵称加入房间并获得座位。
- 四名玩家准备后开始半庄。
- WebSocket 双向通信。
- 服务端权威执行所有游戏行动。
- 玩家视角的隐藏信息过滤。
- 摸牌、出牌、立直、吃、碰、杠、荣和、摸和及流局。
- 行动超时处理。
- 连接断开后保留座位并允许重连。
- 对局结束时向所有玩家发送结算结果。
- 至少支持一台 Linux 服务器部署。

### 暂不实现

- 注册、登录和复杂账户验证。
- 自动匹配、好友列表、排位和积分系统。
- 数据库战绩和长期用户资料。
- 观战、聊天、商店和皮肤。
- 多服务器水平扩展。
- 移动端专用 UI。

## 5. 分阶段实施计划

### 阶段 0：基线确认

目标：确保现有规则引擎是后续网络化的稳定基线。

工作项：

- 运行 workspace 测试和 `riichi-engine` 半庄测试。
- 确认现有协议序列化行为。
- 为非法行动、超时、响应窗口和结算补充回归测试。
- 明确规则版本和默认规则配置。

验收标准：现有本地 AI 对局可以稳定结束，测试结果可重复定位。

### 阶段 1：独立 GameSession

目标：把当前 `GameSession` 封装为与传输方式无关的单房间游戏会话。

工作项：

- 将内部 channel 消息和网络消息继续隔离。
- 引入 `GameSession`，一个实例对应一个房间的一局游戏。
- 保证一个 GameSession 只由一个异步任务顺序修改 `GameState`。
- 抽象玩家命令输入和玩家事件输出。
- 保留现有终端客户端作为适配器。

建议接口：

```rust
pub struct GameSession {
    pub game: GameState,
    pub command_rx: Receiver<PlayerCommand>,
    pub outputs: [PlayerOutput; 4],
}
```

验收标准：不使用 WebSocket 也能通过适配器启动一局完整半庄。

### 阶段 2：协议可靠性

目标：让客户端可以识别丢包、重复提交和过期状态。

工作项：

- 为客户端命令增加 `command_id`。
- 为服务端事件增加递增 `seq`。
- 增加协议版本号。
- 增加完整 `StateSnapshot`。
- 增加命令接受和拒绝消息。
- 增加未知事件的兼容策略。
- 明确每条消息的玩家视角和隐藏字段规则。

建议消息外壳：

```rust
pub struct ClientEnvelope {
    pub protocol_version: u16,
    pub command_id: u64,
    pub expected_seq: u64,
    pub body: ClientMessage,
}

pub struct ServerEnvelope {
    pub protocol_version: u16,
    pub seq: u64,
    pub body: ServerMessage,
}
```

验收标准：客户端从快照开始接收事件；发现序号缺失时可以请求或等待新的完整快照。

### 阶段 3：房间和连接管理

目标：支持朋友通过房间码进入同一局游戏。

工作项：

- `RoomManager`：创建、查找、关闭房间。
- `Room`：玩家、座位、准备状态、游戏状态。
- 玩家昵称和临时连接 token。
- 加入、离开、准备、取消准备。
- 房间人数和状态广播。
- 游戏开始后禁止新玩家替换已有座位。
- 断线玩家保留座位一段时间。

第一版可使用内存结构，不引入数据库。

验收标准：四个独立客户端可以通过房间码加入、准备并开始游戏。

### 阶段 4：HTTP 和 WebSocket 服务

目标：建立真正独立的后端进程。

工作项：

- 使用 Axum/Tokio 提供 HTTP 和 WebSocket 入口。
- WebSocket 鉴权暂时使用房间码、昵称和临时 token。
- 实现 ping/pong 心跳。
- 将连接读写拆分为独立任务。
- 将客户端消息转换为带服务端身份的内部命令。
- 禁止从网络消息中信任 `PlayerId`。
- 连接关闭时通知对应 Room/GameSession。

验收标准：终端客户端或测试脚本可以通过网络完成一局游戏。

### 阶段 5：Tauri 客户端与 Web UI

目标：以 Tauri 为主要桌面客户端，使用 Web 技术实现完整可玩的 UI；浏览器模式只作为开发预览和联调入口。

建议组成：

```text
tauri-app/
├── src/              Web UI 与状态管理
├── src-tauri/        窗口、配置、日志和系统能力
└── package.json

web-client/
├── transport/       WebSocket 和重连
├── protocol/        协议类型
├── store/           房间和牌局状态
├── screens/         加入房间、等待、牌桌、结算
└── components/      手牌、牌河、操作按钮、玩家信息
```

当前实现目录为 `tauri-app/`，其中 `src/protocol.ts` 和 `src/transport.ts` 负责协议类型与 HTTP/WebSocket 传输，`src/main.ts` 当前承载房间大厅和牌桌基础界面。浏览器预览使用 Vite，默认连接本地 `riichi-server`。

工作项：

- 输入服务器地址、昵称和房间码。
- 等待房间和准备界面。
- 牌桌布局和四家信息。
- 自己的手牌和摸牌视觉区分。
- 出牌、立直、吃、碰、杠、胡、取消操作。
- 行动提示和倒计时。
- 错误提示和连接状态。
- 结算界面。
- Tauri 开发环境下支持浏览器预览和桌面窗口运行。
- 客户端配置服务器地址，不把服务器或麻将规则引擎嵌入客户端。

开发验证命令：

```text
cd tauri-app
npm run build
npm run tauri dev
```

Linux 下执行 `npm run tauri dev` 还需要系统安装 Tauri WebView 所需的 `pkg-config`、GTK、WebKitGTK、Pango、Cairo 等开发库；缺少这些库时可以先使用 `npm run build` 和浏览器预览完成前端联调。

验收标准：四个 Tauri 客户端可以不依赖终端正常打完半庄；同一前端可在浏览器预览模式下联调。

### 阶段 6：重连、托管和稳定性

目标：达到“朋友正常游玩”的可靠程度。

工作项：

- 重连 token 和座位恢复。
- 重连后发送玩家视角完整快照。
- 重连后重新发送当前行动请求。
- 行动超时后自动 Pass 或执行托管策略。
- 重复命令去重。
- 服务端拒绝过期 `expected_seq`。
- 前端网络断开提示和自动重连。
- 房间结束和异常退出清理。

验收标准：模拟断网、重复点击、刷新页面后，对局不会泄露手牌或进入不可恢复状态。

### 阶段 7：Tauri 打包和生产部署

目标：将阶段 5 的 Tauri 客户端打包发布，并完成服务器生产部署。

工作项：

- 固化 Tauri 应用目录、图标和生产构建配置。
- 配置开发环境下的本地前端服务和生产环境静态资源。
- 使用 Tauri 处理窗口、配置、日志和系统通知。
- 配置 Windows/Linux 构建。
- 服务器启用 HTTPS/WSS。
- 增加启动配置和服务器地址设置。

验收标准：朋友可以安装桌面客户端，输入服务器地址和房间码后开始游戏。

## 6. 测试计划

### 引擎测试

- 每种行动的合法性和非法性。
- 立直、暗杠、加杠、抢杠。
- 吃碰杠响应优先级。
- 多家荣和和流局。
- 点数、排名和终局条件。

### 协议测试

- 序列化和反序列化。
- 玩家视角隐藏信息。
- 事件序号连续性。
- 重复命令和过期命令。
- 未知字段和未来版本兼容性。

### 服务测试

- 四人加入和准备。
- 房间满员和重复加入。
- 连接断开和重连。
- WebSocket 心跳。
- 多房间并行运行。
- 操作超时。

### 手工验收

- 四个浏览器预览窗口完成一局半庄（当前阶段 5 联调目标）。
- 四个 Tauri 客户端完成一局半庄（待原生依赖和完整操作 UI 完成后验收）。
- 一名玩家中途刷新并重连。
- 多名玩家同时响应荣和/吃碰杠。
- 非法客户端无法改变牌局结果。

## 8. 主要风险与应对

### 状态同步错误

使用服务端权威状态、事件序号和完整快照。客户端只渲染服务端视图，不自行推导规则结果。

### 隐藏信息泄露

所有服务端事件都必须先经过玩家视角投影。协议测试需要明确检查其他玩家手牌、摸牌和牌山信息不会被发送。

### 响应窗口竞态

一个 GameSession 内串行处理命令，由引擎统一决定响应优先级和超时结果。

### 客户端作弊

客户端只提交行动意图。服务器自己保存牌山、手牌、随机数和结算过程。

### 过早引入复杂基础设施

第一版使用单体服务器和内存房间。等核心玩法稳定后，再考虑数据库、Redis 和多实例部署。

## 10. 完成定义

当以下条件全部满足时，可以认为 MVP 完成：

- 四名朋友可以通过房间码加入同一房间。
- 四人准备后可以开始半庄。
- 所有麻将行动均由服务端验证和执行。
- 任意玩家只能看到自己应该看到的信息。
- 客户端可以正确显示每次摸牌、出牌、鸣牌、和牌和结算。
- 玩家短暂断线后可以恢复座位和当前视角状态。
- 非法行动、重复行动和过期行动不会破坏牌局。
- 一局结束后服务器和所有客户端拥有一致的最终结果。
- 可以在一台普通 Linux 服务器上启动并供朋友连接。
