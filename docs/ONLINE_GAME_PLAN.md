# 联网日麻游戏工程计划

## 1. 项目目标

将当前以 Rust 本地进程和 channel 驱动的日麻项目，逐步发展为一个可以和朋友稳定游玩的联网半庄游戏：

- 玩家通过房间码加入房间。
- 四名玩家可以准备并开始一局半庄。
- 独立服务器作为唯一权威，负责牌山、状态、合法性和结算。
- 客户端通过 WebSocket 接收状态和事件、提交玩家行动。
- 首先支持浏览器 Web 客户端，稳定后再用 Tauri 打包为桌面应用。
- 暂不实现复杂账户、好友、匹配、排位和持久化战绩系统。

## 2. 当前基础与边界

当前项目已经具备以下核心能力：

- `riichi-core`：牌、手牌、牌山和玩家等基础领域类型。
- `riichi-logic`：向听、和牌、役种、符数、点数及牌效分析。
- `riichi-engine`：`GameState`、行动合法性、回合推进、响应窗口和结算。
- `riichi-proto`：客户端/服务端消息、玩家视角状态和隐藏信息过滤。
- `riichi-server`：网络、房间和连接边界；当前保留旧 CLI 入口作为迁移兼容层。
- `riichi-debug`：终端调试客户端，提供本地 AI 填位模式。
- `riichi-session`：传输无关的单房间游戏会话，承载 GameLoop、内部命令和事件。

当前主要缺口是：独立网络入口、房间管理、连接会话、协议可靠性、Web UI、断线处理和部署流程。

## 3. 目标架构

```text
tauri-app / web-client
        │ WebSocket + HTTP
        ▼
riichi-server
  ├── transport       HTTP、WebSocket、心跳
  ├── session         连接、座位、重连
  ├── room            房间码、入座、准备
  ├── application     房间和游戏服务编排
  └── game            GameSession、事件分发、超时
        │
        ▼
riichi-engine
        │
        ├── riichi-core
        └── riichi-logic
```

依赖方向必须保持单向：

```text
tauri-app  ───────► riichi-proto
web-client ───────► 协议定义
riichi-server ────► riichi-engine + riichi-proto
riichi-engine ────► riichi-core + riichi-logic
```

客户端不应依赖 `riichi-server` 或 `riichi-engine`。`riichi-debug` 通过 `riichi-session` 运行本地模式，未来网络模式只依赖协议和传输适配器。

## 4. MVP 范围

### 必须完成

- 创建房间并生成短房间码。
- 玩家以昵称加入房间并获得座位。
- 四名玩家准备后开始半庄。
- WebSocket 双向通信。
- 服务端权威执行所有游戏行动。
- 玩家视角的隐藏信息过滤。
- 摸牌、出牌、立直、吃、碰、杠、荣和、摸和及流局。
- 行动超时处理。
- 连接断开后保留座位并允许重连。
- 对局结束时向所有玩家发送结算结果。
- 至少支持一台 Linux 服务器部署。

### 暂不实现

- 注册、登录和复杂账户验证。
- 自动匹配、好友列表、排位和积分系统。
- 数据库战绩和长期用户资料。
- 观战、聊天、商店和皮肤。
- 多服务器水平扩展。
- 移动端专用 UI。

## 5. 分阶段实施计划

### 阶段 0：基线确认

目标：确保现有规则引擎是后续网络化的稳定基线。

工作项：

- 运行 workspace 测试和 `riichi-engine` 半庄测试。
- 确认现有协议序列化行为。
- 为非法行动、超时、响应窗口和结算补充回归测试。
- 明确规则版本和默认规则配置。

验收标准：现有本地 AI 对局可以稳定结束，测试结果可重复定位。

### 阶段 1：独立 GameSession

目标：把当前 `GameLoop` 封装为与传输方式无关的单房间游戏会话。

工作项：

- 将内部 channel 消息和网络消息继续隔离。
- 引入 `GameSession`，一个实例对应一个房间的一局游戏。
- 保证一个 GameSession 只由一个异步任务顺序修改 `GameState`。
- 抽象玩家命令输入和玩家事件输出。
- 保留现有终端客户端作为适配器。

建议接口：

```rust
pub struct GameSession {
    pub game: GameState,
    pub command_rx: Receiver<PlayerCommand>,
    pub outputs: [PlayerOutput; 4],
}
```

验收标准：不使用 WebSocket 也能通过适配器启动一局完整半庄。

### 阶段 2：协议可靠性

目标：让客户端可以识别丢包、重复提交和过期状态。

工作项：

- 为客户端命令增加 `command_id`。
- 为服务端事件增加递增 `seq`。
- 增加协议版本号。
- 增加完整 `StateSnapshot`。
- 增加命令接受和拒绝消息。
- 增加未知事件的兼容策略。
- 明确每条消息的玩家视角和隐藏字段规则。

建议消息外壳：

```rust
pub struct ClientEnvelope {
    pub protocol_version: u16,
    pub command_id: u64,
    pub expected_seq: u64,
    pub body: ClientMessage,
}

pub struct ServerEnvelope {
    pub protocol_version: u16,
    pub seq: u64,
    pub body: ServerMessage,
}
```

验收标准：客户端从快照开始接收事件；发现序号缺失时可以请求或等待新的完整快照。

### 阶段 3：房间和连接管理

目标：支持朋友通过房间码进入同一局游戏。

工作项：

- `RoomManager`：创建、查找、关闭房间。
- `Room`：玩家、座位、准备状态、游戏状态。
- 玩家昵称和临时连接 token。
- 加入、离开、准备、取消准备。
- 房间人数和状态广播。
- 游戏开始后禁止新玩家替换已有座位。
- 断线玩家保留座位一段时间。

第一版可使用内存结构，不引入数据库。

验收标准：四个独立客户端可以通过房间码加入、准备并开始游戏。

### 阶段 4：HTTP 和 WebSocket 服务

目标：建立真正独立的后端进程。

工作项：

- 使用 Axum/Tokio 提供 HTTP 和 WebSocket 入口。
- WebSocket 鉴权暂时使用房间码、昵称和临时 token。
- 实现 ping/pong 心跳。
- 将连接读写拆分为独立任务。
- 将客户端消息转换为带服务端身份的内部命令。
- 禁止从网络消息中信任 `PlayerId`。
- 连接关闭时通知对应 Room/GameSession。

验收标准：终端客户端或测试脚本可以通过网络完成一局游戏。

### 阶段 5：Web 客户端

目标：先在浏览器中实现完整可玩的 UI。

建议组成：

```text
web-client/
├── transport/       WebSocket 和重连
├── protocol/        协议类型
├── store/           房间和牌局状态
├── screens/         加入房间、等待、牌桌、结算
└── components/      手牌、牌河、操作按钮、玩家信息
```

工作项：

- 输入服务器地址、昵称和房间码。
- 等待房间和准备界面。
- 牌桌布局和四家信息。
- 自己的手牌和摸牌视觉区分。
- 出牌、立直、吃、碰、杠、胡、取消操作。
- 行动提示和倒计时。
- 错误提示和连接状态。
- 结算界面。

验收标准：四人不依赖终端，可以正常打完半庄。

### 阶段 6：重连、托管和稳定性

目标：达到“朋友正常游玩”的可靠程度。

工作项：

- 重连 token 和座位恢复。
- 重连后发送玩家视角完整快照。
- 重连后重新发送当前行动请求。
- 行动超时后自动 Pass 或执行托管策略。
- 重复命令去重。
- 服务端拒绝过期 `expected_seq`。
- 前端网络断开提示和自动重连。
- 房间结束和异常退出清理。

验收标准：模拟断网、重复点击、刷新页面后，对局不会泄露手牌或进入不可恢复状态。

### 阶段 7：Tauri 打包和部署

目标：将稳定的 Web 客户端变成桌面应用。

工作项：

- 新增 Tauri 应用目录，不把服务器嵌入客户端。
- 配置开发环境下的本地 Web 服务和生产环境静态资源。
- 使用 Tauri 处理窗口、配置、日志和系统通知。
- 配置 Windows/Linux 构建。
- 服务器启用 HTTPS/WSS。
- 增加启动配置和服务器地址设置。

验收标准：朋友可以安装桌面客户端，输入服务器地址和房间码后开始游戏。

## 6. 测试计划

### 引擎测试

- 每种行动的合法性和非法性。
- 立直、暗杠、加杠、抢杠。
- 吃碰杠响应优先级。
- 多家荣和和流局。
- 点数、排名和终局条件。

### 协议测试

- 序列化和反序列化。
- 玩家视角隐藏信息。
- 事件序号连续性。
- 重复命令和过期命令。
- 未知字段和未来版本兼容性。

### 服务测试

- 四人加入和准备。
- 房间满员和重复加入。
- 连接断开和重连。
- WebSocket 心跳。
- 多房间并行运行。
- 操作超时。

### 手工验收

- 四个浏览器窗口完成一局半庄。
- 四个 Tauri 客户端完成一局半庄。
- 一名玩家中途刷新并重连。
- 多名玩家同时响应荣和/吃碰杠。
- 非法客户端无法改变牌局结果。

## 7. 预计工作量

以已有规则引擎为前提，且采用内存房间和临时身份：

| --- | ---: | ---: |
| 基线和 GameSession | 3–7 天 | 1–2 周 |
| 协议和 WebSocket | 1–2 周 | 2–4 周 |
| 房间和连接管理 | 1–2 周 | 2–4 周 |

## 8. 主要风险与应对

### 状态同步错误

使用服务端权威状态、事件序号和完整快照。客户端只渲染服务端视图，不自行推导规则结果。

### 隐藏信息泄露

所有服务端事件都必须先经过玩家视角投影。协议测试需要明确检查其他玩家手牌、摸牌和牌山信息不会被发送。

### 响应窗口竞态

一个 GameSession 内串行处理命令，由引擎统一决定响应优先级和超时结果。

### 客户端作弊

客户端只提交行动意图。服务器自己保存牌山、手牌、随机数和结算过程。

### 过早引入复杂基础设施

第一版使用单体服务器和内存房间。等核心玩法稳定后，再考虑数据库、Redis 和多实例部署。

## 9. 第一轮开发任务

建议下一轮只实现以下范围：

1. 将 `GameLoop` 重命名或封装为 `GameSession`。
2. 抽象与传输无关的玩家命令和玩家输出。
3. 在 `riichi-server` 中增加一个最小 WebSocket 服务。
4. 用一个网络终端客户端连接服务器。
5. 增加四人房间和准备状态。
6. 增加协议事件序号和完整状态快照。
7. 为重连和隐藏信息补充测试。

第一轮完成后，再开始 Web/Tauri UI，避免 UI 开发建立在尚未稳定的网络协议上。

## 10. 完成定义

当以下条件全部满足时，可以认为 MVP 完成：

- 四名朋友可以通过房间码加入同一房间。
- 四人准备后可以开始半庄。
- 所有麻将行动均由服务端验证和执行。
- 任意玩家只能看到自己应该看到的信息。
- 客户端可以正确显示每次摸牌、出牌、鸣牌、和牌和结算。
- 玩家短暂断线后可以恢复座位和当前视角状态。
- 非法行动、重复行动和过期行动不会破坏牌局。
- 一局结束后服务器和所有客户端拥有一致的最终结果。
- 可以在一台普通 Linux 服务器上启动并供朋友连接。
