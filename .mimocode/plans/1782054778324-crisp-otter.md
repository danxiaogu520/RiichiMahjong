# 重构计划：去除 Server 中人类/AI 的代码区分

## 核心原则

Server 不知道也不关心谁是人类谁是 AI。Server 只做一件事：
1. 游戏需要行动时 → 向当前玩家发送请求 → 等待响应 → 执行
2. 游戏需要副露响应时 → 向相关玩家发送请求 → 等待响应 → 执行

人类/AI 的区别仅在于"谁在 channel 的另一端"。

## 架构变更

### 1. 统一玩家接口：`PlayerHandle`

每个玩家在 Server 看来都是一个 `PlayerHandle`，持有发送请求的 tx 和接收响应的 rx：

```rust
// riichi-server/src/channel.rs

pub struct PlayerHandle {
    pub id: PlayerId,
    pub event_tx: mpsc::Sender<ServerEvent>,   // 发给该玩家的事件
    pub action_rx: mpsc::Receiver<PlayerAction>, // 该玩家的响应
}
```

### 2. `PlayerAction` 携带 PlayerId

当前 `PlayerAction` 不携带玩家 ID，Server 假设所有响应来自 `PlayerId(0)`。必须加上：

```rust
#[derive(Debug, Clone)]
pub enum PlayerAction {
    TurnAction { player: PlayerId, action: TurnActionMsg },
    CallResponse { player: PlayerId, response: CallResponseMsg },
}
```

### 3. GameLoop 持有 4 个 PlayerHandle

```rust
pub struct GameLoop {
    pub game: GameState,
    pub calc: ShantenCalculator,
    pub rng: StdRng,
    pub players: [PlayerHandle; 4],  // 替代单一的 event_tx + action_rx
}
```

### 4. 统一的行动/响应处理

**删除**所有 `if player == PlayerId(0)` 分支。替换为：

```rust
// 向玩家发送行动请求并等待响应
async fn request_action(&mut self, player: PlayerId) -> Option<TurnActionMsg> {
    let handle = &self.players[player.0];
    let _ = handle.event_tx.send(ServerEvent::ActionRequired { ... }).await;
    // 从统一的响应 channel 接收
    // ...（见下方多路复用设计）
}

// 向玩家发送副露请求并等待响应
async fn request_call_response(&mut self, player: PlayerId, options: Vec<CallOption>) -> Option<CallResponseMsg> {
    let handle = &self.players[player.0];
    let _ = handle.event_tx.send(ServerEvent::CallRequired { options }).await;
    // ...
}
```

### 5. 多路复用响应接收

问题：4 个玩家各有自己的 `action_rx`，Server 需要同时监听多个 channel。

方案：用 `tokio::sync::mpsc` + `tokio::select!` 或合并为单一 channel：

**方案 A（推荐）：合并为带 ID 的单一 channel**

```rust
pub struct GameLoop {
    pub game: GameState,
    pub calc: ShantenCalculator,
    pub rng: StdRng,
    pub event_txs: [mpsc::Sender<ServerEvent>; 4],
    pub action_rx: mpsc::Receiver<(PlayerId, PlayerAction)>,  // 合并的响应 channel
}
```

每个 PlayerHandle 的 `action_tx` 都连接到同一个合并 channel，发送时附带自己的 PlayerId。

**方案 B：tokio::select! 监听 4 个 channel**

更灵活但更复杂。暂不采用。

### 6. 创建 PlayerHandle 的工厂函数

```rust
// riichi-server/src/channel.rs

/// 创建一对连接：返回 (PlayerHandle, ClientHandle)
/// PlayerHandle 给 Server 用，ClientHandle 给客户端（人类UI 或 AI）用
pub fn create_player_pair(id: PlayerId) -> (PlayerHandle, ClientHandle) {
    let (event_tx, event_rx) = mpsc::channel(64);
    let (action_tx, action_rx) = mpsc::channel(64);
    (
        PlayerHandle { id, event_tx, action_rx },
        ClientHandle { id, event_rx, action_tx },
    )
}

pub struct ClientHandle {
    pub id: PlayerId,
    pub event_rx: mpsc::Receiver<ServerEvent>,
    pub action_tx: mpsc::Sender<(PlayerId, PlayerAction)>,
}
```

### 7. AI 客户端

AI 客户端和人类客户端一样，持有 `ClientHandle`，循环接收事件并发送决策：

```rust
// riichi-ai/src/client.rs (新增)

pub async fn run_ai_client(mut handle: ClientHandle) {
    let mut calc = ShantenCalculator::new();
    while let Some(event) = handle.event_rx.recv().await {
        match event {
            ServerEvent::ActionRequired { can_tsumo, can_riichi } => {
                let action = ai_decide_turn(&handle.id, can_tsumo, can_riichi, &mut calc);
                let _ = handle.action_tx.send((handle.id, action)).await;
            }
            ServerEvent::CallRequired { options } => {
                let response = ai_decide_call(&handle.id, &options);
                let _ = handle.action_tx.send((handle.id, response)).await;
            }
            ServerEvent::GameOver { .. } => break,
            _ => {}
        }
    }
}
```

### 8. 入口 main.rs

```rust
// 创建 4 个玩家对
let (p0_handle, p0_client) = create_player_pair(PlayerId(0));
let (p1_handle, p1_client) = create_player_pair(PlayerId(1));
let (p2_handle, p2_client) = create_player_pair(PlayerId(2));
let (p3_handle, p3_client) = create_player_pair(PlayerId(3));

// 启动 3 个 AI 客户端
tokio::spawn(run_ai_client(p1_client));
tokio::spawn(run_ai_client(p2_client));
tokio::spawn(run_ai_client(p3_client));

// 启动 Server
let mut game_loop = GameLoop::new([p0_handle, p1_handle, p2_handle, p3_handle]);
tokio::spawn(game_loop.run());

// 人类客户端在主线程运行（ratatui）
run_human_client(p0_client).await;
```

## 需要修改的文件

### `riichi-server/src/channel.rs` — 重写
- 新增 `PlayerHandle`、`ClientHandle`、`create_player_pair()`
- `PlayerAction` 增加 `PlayerId` 字段
- `ServerEvent` 增加 `player: PlayerId` 字段（标识发给谁）

### `riichi-server/src/game_loop.rs` — 重写核心逻辑
- `GameLoop` 持有 `[PlayerHandle; 4]` 而非单一 channel
- 删除所有 `PlayerId(0)` 硬编码
- 新增 `request_action(player)` — 向指定玩家发送行动请求
- 新增 `request_call(player, options)` — 向指定玩家发送副露请求
- `run()` 循环统一处理所有玩家
- 副露处理：遍历所有有选项的玩家，逐个请求响应
- 删除 `run_ai_turn()`、`ai_discard()`、`ai_choose_riichi_tile()` 等 AI 逻辑（移到 riichi-ai）

### `riichi-server/src/main.rs` — 重写入口
- 创建 4 个 player pair
- 3 个 AI + 1 个人类

### `riichi-server/src/lib.rs` — 更新导出

### `riichi-ai/src/client.rs` — 新增
- AI 客户端循环：接收事件 → 决策 → 发送响应
- 从 game_loop.rs 迁移过来的 AI 决策逻辑

### `riichi-ai/src/lib.rs` — 新增导出

### `riichi-client/src/app.rs` — 适配
- `PlayerAction` 类型变了（增加了 PlayerId 字段）
- `send_discard()` 等方法需要附带 PlayerId

### `riichi-client/src/main.rs` — 适配
- 使用 `ClientHandle` 而非独立的 channel
