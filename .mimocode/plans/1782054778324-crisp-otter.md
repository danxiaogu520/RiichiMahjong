# 日麻引擎前后端分离重构方案

## 目标

将当前单体架构重构为前后端分离的模块化架构，核心原则：
- **Engine 只管状态变更**，不嵌入任何计算逻辑
- **Logic 模块承载所有纯算法**（向听、有效牌、改良、算点、判役）
- **AI 模块做复杂决策**（目前简单实现：不副露 + 按牌理打牌）
- **Server 管理游戏实例和房间**，与客户端通信
- **Client 只负责展示和操作**，不涉及计算

---

## Crate 命名映射

| 旧名称 | 新名称 | 职责 |
|--------|--------|------|
| `mahjong-core` | **`riichi-core`** | 纯数据结构（扩展） |
| `mahjong-yaku` | **删除**（合并入 riichi-logic） | — |
| *(新)* | **`riichi-logic`** | 纯算法：向听、判役、算点、有效牌、宝牌 |
| `mahjong-ai` | **`riichi-ai`** | 决策层：打牌选择、副露决策、立直决策 |
| `mahjong-engine` | **`riichi-engine`** | 纯状态机（大幅瘦身） |
| `mahjong-server` | **`riichi-server`** | 游戏服务：房间管理、AI 控制、与客户端通信 |
| `mahjong-tui` | **`riichi-client`** | 客户端：纯展示与操作 |
| *(新)* | **`riichi-proto`** | 通信协议：消息类型定义 |

## 依赖图

```
riichi-proto   (standalone: serde + tokio)
     ↑
riichi-core    (tile, hand, meld, wall, player, game_types, player_state)
     ↑
riichi-logic   (shanten, yaku, fu, scoring, dora, analysis, win_check, acceptance)
     ↑                  ↑
riichi-ai                riichi-engine   (纯状态机, 仅依赖 core)
     ↑                  ↑
     └── riichi-server ← riichi-proto
             ↑
        riichi-client ← riichi-proto
```

关键约束：
- `riichi-engine` 仅依赖 `riichi-core`（+ rand, serde, thiserror）
- `riichi-ai` 依赖 `riichi-core` + `riichi-logic`
- `riichi-client` 依赖 `riichi-core` + `riichi-proto`，**不依赖** logic/ai/engine
- `riichi-server` 依赖 core + engine + logic + ai + proto

### 通信方式
- **tokio::mpsc channel** 实现进程内通信（单进程多线程）
- Server 持有 `Sender<ServerMessage>` 发送给 Client
- Client 持有 `Sender<ClientMessage>` 发送给 Server
- 未来可替换为 TCP/WebSocket（proto 层已定义好消息格式）

### 分析信息
- **Server 负责计算**分析信息（向听数/有效牌/改良牌），通过 `GameStateView.analysis` 传递给 Client
- Client 不依赖 `riichi-logic`，纯展示层

## Phase 1: 创建 `riichi-proto` crate

**新增** `crates/riichi-proto/`，定义客户端-服务端通信协议。

### 模块结构
```
src/
  messages.rs   — 所有消息类型
  lib.rs
```

### 消息类型

```rust
// 客户端 → 服务端
enum ClientMessage {
    JoinRoom { room_id: String },
    TurnAction { action: TurnActionPayload },
    CallResponse { action: CallResponsePayload },
    Ready,
    LeaveRoom,
}

// 服务端 → 客户端
enum ServerMessage {
    RoomJoined { room_id: String, player_id: PlayerId },
    StateUpdate(GameStateView),
    Event(GameEvent),
    ActionRequired(ActionRequest),
    CallRequired(CallRequest),
    RoundResult(RoundResult),
    GameOver { scores: [i32; 4] },
    Error(String),
}

// 玩家视角（隐藏他人手牌信息）
struct GameStateView {
    players: [PlayerView; 4],
    wind, round, honba, riichi_sticks,
    current_player, drawn_tile: Option<Tile>,
    dora, remaining_tiles, phase, recent_events,
    analysis: Option<AnalysisInfo>,  // Server 计算的分析信息
}

struct AnalysisInfo {
    discard_options: Vec<DiscardOptionView>,
    acceptance: Vec<AcceptanceInfoView>,
    improvement: Vec<AcceptanceInfoView>,
    current_shanten: i8,
}

struct PlayerView {
    id, hand: Option<Vec<Tile>>, hand_count, points, wind,
    discards, melds, is_riichi, riichi_declaration_tile,
}
```

**依赖**: `riichi-core` + `serde` + `tokio`

## Phase 2: 将共享类型下沉到 `riichi-core`

将 engine 中被多个 crate 共享的类型移入 core。

### 新增文件

| 新文件 | 从何处移入 | 类型 |
|--------|-----------|------|
| `core/src/game_types.rs` | `engine/src/action.rs` + `engine/src/game.rs` | `GamePhase`, `GameEvent`, `RoundEndReason`, `TurnAction`, `ResponseAction`, `CallOption`, `CallType`, `GameError` |
| `core/src/player_state.rs` | `engine/src/player.rs` | `Player` struct, `FuritenState` struct |
| `core/src/player.rs` | `engine/src/player.rs` | `wind_from_index()`, `next_wind()`, `wind_display()` |

### 迁移要点
- `GameEvent`, `RoundEndReason` 等已 derive `Serialize/Deserialize`，直接移入
- `Player` 依赖 `Hand`, `Meld`, `PlayerId`, `Tile` — 已在 core 中
- engine 改为 re-export: `pub use riichi_core::game_types::*`
- 删除 `engine/src/action.rs` 和 `engine/src/player.rs`
- 更新 server/tui 的 import 路径

---

## Phase 3: Wall 整合

将 engine 中内联的牌山管理整合到 `core/src/wall.rs`。

### 变更
- `GameState` 的 `wall: Vec<Tile>` + `current_index: usize` → 改为持有 `Wall`
- `Wall` 新增方法：`remaining()`, `draw()`, `draw_rinshan()`, `dora_indicator()`, `kan_count()`
- engine 的 `game.rs` 中 `draw()`, `draw_rinshan()`, `remaining_tiles()` 改为调用 `Wall`
- `RINSHAN=135`, `DORA_INDICATOR=131`, `HAITEI=121` 常量移入 `Wall` 内部

## Phase 4: 创建 `riichi-logic`（合并 yaku + ai 算法）

**新增** `crates/riichi-logic/`，将所有纯算法集中到此 crate。

### 模块结构
```
src/
  types.rs          ← yaku/types.rs (YakuName, WinContext, TileCounts, etc.)
  analysis.rs       ← yaku/analysis.rs (is_winning, decompose_hand, analyze_wait_tiles)
  win_check.rs      ← yaku/win_check.rs (check_win, detect_yaku)
  fu.rs             ← yaku/fu.rs (calculate_fu)
  scoring.rs        ← yaku/scoring.rs (calculate_points)
  dora.rs           ← yaku/dora.rs (calculate_dora, dora_from_indicator)
  shanten.rs        ← ai/shanten.rs (ShantenCalculator + lookup tables)
  acceptance.rs     ← ai/analysis.rs (analyze_discard, analyze_acceptance, DiscardOption, VisibleTiles)
  lib.rs
data/
  index_s.bin       ← ai/data/index_s.bin
  index_h.bin       ← ai/data/index_h.bin
```

**依赖**: `riichi-core` only（无 rand）

**注意**: 删除所有 `#[cfg(test)]` 模块和测试代码。

## Phase 6: 重构 `riichi-ai` 为纯决策层

算法部分已移入 `riichi-logic`，`riichi-ai` 变为纯决策。

### 新模块结构
```
src/
  discard.rs          — choose_discard() 调用 logic 的 analyze_discard
  call_decision.rs    — decide_call() 碰/吃/杠/荣和决策
  riichi_decision.rs  — decide_riichi() 是否立直 + 宣言牌选择
  lib.rs
```

### API
```rust
// discard.rs
pub fn choose_discard(...) -> DiscardOption;

// call_decision.rs
pub fn decide_call(
    player: PlayerId, options: &[CallOption], hand: &Hand,
    melds: &[Meld], visible: &VisibleTiles,
    calculator: &mut ShantenCalculator,
) -> Option<ResponseAction>;

// riichi_decision.rs
pub fn decide_riichi(
    player: PlayerId, hand: &[Tile], drawn_tile: Tile,
    tenpai_discards: &[Tile], visible: &VisibleTiles,
    calculator: &mut ShantenCalculator,
) -> Option<Tile>;
```

**依赖**: `riichi-core` + `riichi-logic`

**注意**: 删除所有测试代码。

## Phase 7: 解耦 Engine — 删除计算方法

从 `GameState` 中删除以下方法：

| 删除的方法 | 替代方案 |
|-----------|---------|
| `is_tenpai()` | `ShantenCalculator::lookup() == 0` |
| `get_tenpai_discard_options()` | `logic::analyze_discard()` 后筛选 shanten=0 |
| `can_tenpai()` | 同上 |
| `get_waiting_tiles()` | `logic::analyze_wait_tiles()` |
| `check_tsumo()` | 外部组合 `analyze_wait_tiles` + `check_win` |
| `get_riichi_ankan_options()` | 外部模拟暗杠 + 听牌对比 |
| `check_win()` / `check_win_with_hand()` | 外部调用 `logic::check_win()` |
| `get_waiting_tile_types()` | 外部调用 `analyze_wait_tiles()` |

### 改造保留方法的签名

```rust
// update_discard_furiten — 改为接收外部听牌信息
pub fn update_discard_furiten(
    &mut self, player: PlayerId,
    all_waiting: &HashMap<PlayerId, HashSet<TileType>>,
);

// resolve_exhaustive_draw — 改为接收外部听牌状态
pub fn resolve_exhaustive_draw_with_tenpai(&mut self, tenpai: [bool; 4]);

// execute_action(RiichiDiscard) — 移除内部听牌验证，调用方保证合法性
```

**结果**: engine 不再 import `riichi_logic` 或 `riichi_ai`，依赖仅 `riichi-core` + rand + serde + thiserror。

**注意**: 删除所有测试代码。

## Phase 9: 重命名 mahjong-tui → riichi-client + 重构

- `crates/mahjong-tui/` → `crates/riichi-client/`
- 移除对 `riichi-logic` 和 `riichi-ai` 的直接依赖
- 客户端通过 tokio mpsc channel 接收 `ServerMessage`，发送 `ClientMessage`
- 分析面板信息由 Server 计算后通过 `GameStateView.analysis` 传递

### 客户端结构
```
src/
  app.rs           — App struct，持有 tx/rx channel
  input.rs         — 键盘输入处理
  ui.rs            — ratatui 渲染
  widgets/
    board.rs
    status.rs
  main.rs
```

### 进程启动方式
```rust
let (client_tx, server_rx) = tokio::sync::mpsc::channel(32);
let (server_tx, client_rx) = tokio::sync::mpsc::channel(32);

tokio::spawn(async move {
    server.run(server_tx, server_rx).await;
});

let mut app = App::new(client_tx, client_rx);
run_app(&mut terminal, &mut app).await;
```

## 风险与缓解

| 风险 | 缓解 |
|------|------|
| 大规模 import 重写出错 | 每步后 `cargo build` 验证，用批量替换 |
| Engine 解耦后调用方需大量适配 | 逐步进行，先删简单方法再改签名 |
| Wall 整合改变配牌行为 | 用 seed-based 测试确保一致（后续自己写测试验证） |
| Client 无法获取分析信息 | Server 计算后通过 proto 的扩展字段传递 |
